Functional vs Object Oriented Programming

May 9, 2014

Functional vs Object Oriented Programming

The biggest difference between functional and object oriented programming is the notion of state. In functional programming, the output of a function depends only on the parameters passed into it. Functions can interact with other functions, but they stand alone, meaning they do not belong to a class. Without the notion of classes in functional programming, there there are no instance variables that can have an affect on the output of a function. A simple exmaple can be illustrated with a character in a video game.

In object-oriented programming, our character would be represented by a class with built-in functions and instance variables that represent the state of the object:

class Character
  def initialize(name, race, weapon, hometown)
    @name = name
    @race = race
    @weapon = weapon
    @hometown = hometown
  end

  def attack
    case @weapon
    when "sword"
      puts "#{@name} Slashes their Enemy!"
    when "magic"
      puts "#{@name} Casts a Bolt of Acid at their Enemy!"
    when "bow"
      puts "#{@name} Fires a Flaming Arrow at their Enemy!"
  end  
end

In this case, your character would be an instance of the Character class, and its state is represented by instance variables. Each character can have a name, a race, a weapon, and a hometown. If you want to use your character to attack an enemy, you just need to call the 'attack' method. Depending on your character's state, it will attack in a certain way - but you don't worry about that. You just tell it to attack, and the complexity of how it attacks is factored out. Changing that part requires updating the state of the object, or updating it's @weapon instance variable.

Functional programming is different. In functioanl programming, you could still create a character and tell it to attack, but you need to be more specific. Since your attack method doesn't belong to your character's class, you need to tell it who is doing the attacking and how they should attack

def attack(character,weapon)
  case weapon
  when "sword"
    puts "#{character} Slashes their Enemy!"
  when "magic"
    puts "#{character} Casts a Bolt of Acid at their Enemy!"
  when "bow"
    puts "#{character} Fires a Flaming Arrow at their Enemy!"
  end
end

While it's sometimes easier for us to visualize real-world problems in object-oriented terms, there is a place for both Fundamental and Object Oriented Programming. OOP works well when there are a fixed number of actions that objects have to do, but objects are continuously being created or updated. OOP allows you to inherit properties from another class, giving programmers the ability to quickly create new types of objects that differ slightly from their predecessors. Fundamental programming works best when you have a small number of objects, but are continuously adding new functaionlity for the same objects. In this case, developers can update a function in one place, as opposed to updating different instance methods that belong to specific types of objects.
